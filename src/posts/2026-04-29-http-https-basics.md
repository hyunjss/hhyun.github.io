## HTTP가 뭔지 한 줄로 말하면?

클라이언트와 서버가 데이터를 주고받는 **약속(프로토콜)** 이에요.

근데 HTTP는 **무상태(Stateless)** 라는 특징이 있어요. 매 요청이 독립적이어서 서버가 이전 요청을 기억을 못해요.

```
클라이언트: "나 아현이야, 로그인했어"
서버:       "...누구세요?" ← 기억 못함
```

그래서 쿠키/세션/JWT 같은 게 필요한 거예요. 나중에 이것도 정리할 예정!

---

## HTTPS = HTTP + 암호화

HTTPS는 HTTP에 **TLS** 를 씌운 거예요.

- **HTTP** → 데이터가 그냥 평문으로 이동해요. 중간에서 누가 보면 다 보여요.
- **HTTPS** → 암호화해서 이동하니까 중간에서 봐도 해독을 못해요.

실무에서 왜 중요하냐면, 배포 환경에서 HTTPS가 아니면 브라우저가 경고를 띄우거나 카메라, 위치 같은 API를 아예 막아버려요. 쿠키의 `Secure` 속성도 HTTPS에서만 제대로 동작하고요.

로컬에서 `http://localhost`는 괜찮은데 실제 배포할 때는 무조건 HTTPS예요.

## TLS 핸드셰이크는 어떻게 동작하나

면접에서 "HTTPS 동작 방식 설명해보세요" 물어보면 이 흐름으로 말하면 돼요.

```
1. 클라이언트 → 서버    "나 이런 암호화 방식 지원해"
2. 서버 → 클라이언트    "그럼 이걸로 하자, 내 인증서야"
3. 클라이언트           인증서 검증 (이 서버 믿을 수 있는 CA인지 확인)
4. 클라이언트 → 서버    암호화 키 교환
5. 이후 통신            암호화된 데이터 주고받기 ✅
```

`net::ERR_CERT_AUTHORITY_INVALID` 에러 본 적 있죠? 저 3번에서 인증서 검증 실패한 거예요. 로컬에서 자체 서명 인증서(self-signed) 쓸 때 자주 만나요.

## HTTP 요청/응답 구조

Network 탭 열면 매일 보는 것들인데 한번 짚고 넘어가요.

**요청(Request)**
```http
GET /api/users HTTP/1.1
Host: api.example.com
Authorization: Bearer eyJhbGci...
Content-Type: application/json
```

**응답(Response)**
```http
HTTP/1.1 200 OK
Content-Type: application/json
Cache-Control: max-age=3600

{ "users": [...] }
```

### 상태 코드

| 코드 | 의미 | 언제 나오냐면 |
|------|------|-----------|
| 200 | 성공 | 일반 조회 |
| 201 | 생성됨 | POST로 리소스 만들었을 때 |
| 204 | 내용 없음 | DELETE 성공, 응답 바디 없을 때 |
| 301 | 영구 이동 | 도메인 변경, SEO에도 영향 있음 |
| 304 | 변경 없음 | 캐시 유효해서 서버가 바디 안 보낼 때 |
| 400 | 잘못된 요청 | 파라미터 누락이나 형식 오류 |
| 401 | 인증 필요 | 로그인 안 됨 |
| 403 | 권한 없음 | 로그인은 됐는데 접근 불가 |
| 404 | 없음 | 리소스 없음 |
| 500 | 서버 오류 | 백엔드 문제 |

401이랑 403 혼동하는 백엔드 개발자 은근 많아요. "로그인이 안 된 건지, 권한이 없는 건지" 구분해서 내려줘야 FE에서 분기 처리가 깔끔해지거든요. 협업할 때 미리 얘기해두면 좋아요.

## HTTP 버전별 차이

면접에서 자주 나오는 주제예요.

```
HTTP/1.1  요청을 하나씩 순서대로 처리해요.
          이미지 10개면 10번 기다려야 해요. (Head-of-line blocking)

HTTP/2    하나의 연결에서 여러 요청을 동시에 처리해요. (멀티플렉싱)
          이미지 10개를 병렬로 받을 수 있고 헤더도 압축해요.

HTTP/3    UDP 기반으로 바뀌어서 연결 속도가 더 빨라요.
          아직 모든 환경에서 지원하진 않아요.
```

Next.js나 Nuxt 같은 프레임워크가 자동으로 HTTP/2를 활용해요. 번들을 잘게 쪼개도 되는 이유 중 하나가 HTTP/2 덕분이에요.

## 면접 답변 정리

**Q. HTTP와 HTTPS의 차이는?**

HTTP는 데이터를 평문으로 전송하고, HTTPS는 TLS를 통해 암호화해서 전송해요. HTTPS는 인증서로 서버 신뢰성을 검증하고 중간자 공격을 방지해요. 실무에서 쿠키 `Secure` 속성이나 브라우저 보안 API가 HTTPS에서만 동작하기 때문에 배포 환경에서는 필수예요.

**Q. HTTP/1.1과 HTTP/2의 차이는?**

HTTP/1.1은 요청을 순차 처리해서 앞 요청이 지연되면 뒤도 막혀요. HTTP/2는 멀티플렉싱으로 하나의 연결에서 여러 요청을 병렬 처리하고 헤더도 압축해서 성능이 크게 개선됐어요.
